### 资源分类体系重构方案
基于现有 `resources.json` 中的资源特征和使用场景，重构后的分类体系更贴合开发者实际使用习惯，兼顾**技术领域**、**使用场景**和**资源类型**三大维度，同时简化层级、优化命名，提升检索效率。

#### 一、核心分类（一级分类）
保留核心领域维度，合并冗余分类，新增高频场景分类：
| 分类ID | 分类名称 | 图标 | 核心定位 |
|--------|----------|------|----------|
| `development` | 开发技术 | 🛠️ | 覆盖前端/后端/跨端等开发全链路工具与框架 |
| `learning` | 学习资源 | 📚 | 编程学习、技能提升、认证考试相关资源 |
| `tools` | 效率工具 | ⚡ | 开发/设计/办公/系统类工具（非技术框架类） |
| `resources` | 素材资源 | 🎨 | 图标、字体、模板、合集等可复用素材 |

#### 二、子分类（二级分类）
按「使用场景+技术属性」拆分，删除冗余、补充高频场景，适配现有资源：

#### 三、重构核心优化点
1. **简化层级**：删除「frontend」这类过窄的一级分类，将前端相关资源归入「开发技术」下的子分类，更符合“开发全链路”认知；
2. **场景化命名**：将「api-backend」改为「api-service」、「exam-cert」改为「certification」，命名更通用易懂；
3. **补充高频场景**：新增「algorithm（算法刷题）」「tech-docs（技术文档）」等子分类，适配开发者高频需求；
4. **合并冗余分类**：将「resource-collections」拆分至「素材资源」下的子分类，避免“合集”与其他分类重叠；
5. **适配现有资源**：所有原有资源均可无缝映射到新分类体系（如下示例）：

#### 四、原有资源映射示例
| 原有资源 | 原分类路径 | 新分类路径 |
|----------|------------|------------|
| Apifox | frontend > api-backend | development > api-service |
| 力扣 | frontend > docs-learning | learning > algorithm |
| Element Plus | frontend > ui-libraries | development > ui-component |
| iconfont | frontend > icons-fonts | resources > icons-fonts |
| Flutter | frontend > frontend-frameworks | development > frontend-framework |
| 掘金 | frontend > docs-learning | development > tech-docs |
| AI工具 | software-tools > ai-tools | tools > ai-tools |

#### 五、扩展建议
1. 后续可根据资源增长，在「开发技术」下新增「backend-framework（后端框架）」「database（数据库）」等子分类；
2. 为子分类增加「标签推荐」（如「ui-component」关联Vue/React/移动端标签），提升检索效率；
3. 新增「热门子分类」标识，优先展示高频使用的子分类（如code-hosting、ui-component、program-tutorial）。

重构后的分类体系既兼容现有资源，又具备扩展性，同时更贴合开发者的使用习惯，降低资源检索成本。
