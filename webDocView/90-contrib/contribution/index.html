贡献指南

约 1996 字大约 7 分钟

2026-02-04

欢迎参与前端学习文档系统的贡献！我们非常感谢每一位贡献者的支持。本指南将帮助你了解如何参与贡献。

贡献方式

报告问题

如果你发现了文档中的错误、遗漏或不清楚的地方，欢迎通过以下方式报告：

  1. GitHub Issues：在 GitHub 仓库中创建 Issue
  2. 描述问题：详细描述你发现的问题
  3. 提供示例：如果可能，提供代码示例或截图
  4. 建议改进：提出你的改进建议

提交文档

如果你希望直接提交文档改进，请按照以下步骤操作：

  1. Fork 仓库：点击 GitHub 仓库右上角的 Fork 按钮
  2. 克隆仓库：将 Fork 的仓库克隆到本地
  3. 创建分支：基于 main 分支创建新的特性分支
  4. 修改文档：按照文档模板修改或新增文档
  5. 提交更改：提交你的更改
  6. 推送分支：将分支推送到你的 Fork 仓库
  7. 发起 PR：在 GitHub 上发起 Pull Request

代码贡献

如果你希望贡献代码（如示例代码、工具脚本等），请确保：

  1. 代码风格：遵循项目的代码风格规范
  2. 代码质量：代码应该清晰、可读、可维护
  3. 代码注释：添加必要的注释说明
  4. 测试验证：确保代码能够正常运行

开发流程

环境准备

安装依赖

pnpm install

启动开发服务器

pnpm docs:dev

构建文档

pnpm docs:build

分支规范

我们使用以下分支命名规范：

  • main：主分支，只包含稳定的发布版本
  • dev：开发分支，用于集成最新的开发内容
  • feature/xxx：特性分支，用于开发新功能
  • fix/xxx：修复分支，用于修复问题
  • docs/xxx：文档分支，用于更新文档

提交规范

我们使用 Conventional Commits 规范：

提交格式

<type>(<scope>): <subject>

<body>

<footer>

提交类型

  • feat：新功能
  • fix：修复问题
  • docs：文档更新
  • style：代码格式调整
  • refactor：代码重构
  • perf：性能优化
  • test：测试相关
  • chore：构建工具或辅助工具的变动

提交示例

# 新增文档
git commit -m "docs(vue3): 添加响应式系统文档"

# 修复错误
git commit -m "fix(vue3): 修复响应式系统示例代码错误"

# 更新文档
git commit -m "docs: 更新文档模板"

PR 流程

创建 PR

  1. 确保分支是最新的：从 main 分支拉取最新代码
  2. 推送分支：将你的分支推送到 GitHub
  3. 发起 PR：在 GitHub 上创建 Pull Request
  4. 填写 PR 模板：按照 PR 模板填写相关信息

PR 模板

## 变更类型
- [ ] 新增文档
- [ ] 修复错误
- [ ] 更新文档
- [ ] 代码贡献

## 变更内容
简要描述你的变更内容。

## 变更原因
说明为什么需要进行这个变更。

## 测试验证
描述你如何验证这个变更。

## 相关 Issue
关联相关的 Issue（如有）。

## 截图
如果适用，提供截图说明变更效果。

PR 审核

PR 提交后，维护者会进行审核：

  1. 代码审查：检查代码质量和风格
  2. 文档审查：检查文档内容和格式
  3. 测试验证：验证文档的正确性
  4. 合并或反馈：合并 PR 或提出修改建议

PR 合并

PR 审核通过后，维护者会将 PR 合并到主分支：

  1. Squash Merge：将多个提交压缩为一个
  2. 删除分支：删除已合并的特性分支
  3. 更新版本：更新版本号和变更日志

文档规范

文档模板

所有新增文档都必须遵循 文档模板 的格式规范。

文档结构

文档应该包含以下部分：

  • 模块概述：简要介绍文档内容
  • 核心概念：介绍核心概念和术语
  • 实现原理：详细说明实现原理
  • 实践应用：提供实际应用示例
  • 注意事项：列出需要注意的问题
  • 总结：总结核心内容
  • 参考资源：提供相关资源链接

文档格式

  • 使用 Markdown 格式
  • 遵循统一的格式规范
  • 使用清晰的标题层级
  • 提供充分的代码示例
  • 添加必要的注释说明

代码示例

  • 代码示例应该完整可运行
  • 使用正确的语言标识
  • 添加必要的注释
  • 保持代码风格一致

链接规范

  • 使用描述性的链接文本
  • 确保链接可访问
  • 内部链接使用相对路径
  • 外部链接在新标签页打开

审核标准

内容质量

  • 准确性：内容应该准确无误
  • 完整性：内容应该完整全面
  • 清晰性：内容应该清晰易懂
  • 实用性：内容应该实用有用

格式规范

  • 遵循模板：遵循文档模板格式
  • 统一风格：保持统一的格式风格
  • 正确语法：使用正确的 Markdown 语法
  • 规范命名：使用规范的文件和目录命名

代码质量

  • 代码风格：遵循项目的代码风格
  • 代码注释：添加必要的代码注释
  • 代码测试：确保代码能够正常运行
  • 代码优化：代码应该简洁高效

文档质量

  • 文档完整：文档应该完整全面
  • 文档清晰：文档应该清晰易懂
  • 文档准确：文档应该准确无误
  • 文档实用：文档应该实用有用

常见问题

如何开始贡献？

  1. 阅读 文档模板
  2. 选择一个你感兴趣的模块
  3. Fork 仓库并创建分支
  4. 按照模板编写文档
  5. 提交 PR

贡献需要什么技能？

  • 基本的 Markdown 写作能力
  • 对前端技术的理解
  • 良好的沟通能力
  • 团队合作精神

贡献需要多长时间？

这取决于贡献的类型和复杂度：

  • 简单修正：几分钟到几小时
  • 新增文档：几小时到几天
  • 代码贡献：几天到几周

贡献会被接受吗？

只要你的贡献符合以下条件，就会被接受：

  • 内容准确无误
  • 遵循文档规范
  • 通过审核标准
  • 不与现有内容冲突

如何联系维护者？

你可以通过以下方式联系维护者：

  • GitHub Issues：创建 Issue 提问
  • Pull Request：在 PR 中提问
  • Email：发送邮件联系

行为准则

尊重他人

  • 尊重所有贡献者
  • 尊重不同的观点
  • 避免人身攻击
  • 保持友好态度

协作精神

  • 积极参与讨论
  • 提供建设性反馈
  • 帮助其他贡献者
  • 共同改进项目

专业态度

  • 保持专业态度
  • 遵循项目规范
  • 及时响应反馈
  • 按时完成任务

许可证

所有贡献的内容都将遵循项目的开源许可证。当你提交贡献时，你同意你的贡献将按照项目的许可证进行发布。
